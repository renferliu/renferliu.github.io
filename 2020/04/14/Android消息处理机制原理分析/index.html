Android消息处理机制原理分析

发表于 2020-04-14 更新于 2020-05-12 分类于 安卓

Android系统主要通过Handler、Looper、MessageQueue三个类实现Android应用程序的消息处理机制。Handler用来发送消息和处理消息，Looper用来创建消息队列，MessageQueue用来描述消息队列。我们先了解Handler、Looper、MessageQueue这三个类的功能职责，再来研究整个消息处理机制运行的过程及源码执行过程。

Handler、Looper、MessageQueue的功能职责

Handler

介绍

关于Handler的功能，官方是这样描述的：

Handler能够发送以及处理与线程消息队列（即MessageQueue对象）关联的Message和Runnable对象。每个Handler实例都与单一的线程及其消息队列（即MessageQueue对象）有关。当创建一个Handler实例时，其绑定到线程/创建线程的消息队列。同时，Handler将Message和Ruannable对象传递到消息队列，最后再他们从消息队列出来后执行。

用法

Handler主要用法有以下两种：

  1. 安排Message以及Runnable对象在未来的某个时间点执行（详见code-block 1）；
  2. 让代码段切换至不同于当前线程中执行，通常情况在将工作线程中的某个代码块切换至主线程中执行（详见clode-block 2）。

Looper

介绍

该类主要被用于线程运行消息循环。默认情况下线程并没有与之关联的消息循环；如果要为线程创建一个消息循环，需要调用prepare()并运行在需要循环的线程当中，然后调用loop()不断处理消息，直到循环停止为止。

MessageQueue

介绍

MessageQueue是一个底层的容器类，用于装载Looper将要发送的消息。消息无法直接添加至MessageQueue中，而是通过通过与Looper所关联的Handler装载进去。

要弄懂enqueueMessage流程，我们主要分析MessageQueue.class中的enqueueMessage方法便可。之前发送消息的时候都需要随消息传入一个处理时间，这个处理时间便是用于消息进入队列进行排序所用。这个消息队列是根据处理时间由大到小进行排列的。故在插入消息进入消息队列时要根据处理时间进行判断（如38行代码所示）。主要分成以下四种情况：

  1. 目标队列是一个空的消息队列
  2. 插入的消息消息处理时间等于0
  3. 插入的消息的处理时间小于保存在目标消息队列头的消息处理时间
  4. 插入的消息的处理时间大于或等于保存在目标消息队列头的消息处理时间

1-3类情况则执行40-42行代码逻辑，即将插入的消息保存在目标消息队列的头部。

第4类情况则执行47-60行代码逻辑，即根据插入消息的处理时间按从小到大的规则保存在目标队列的合适位置中。

消息被插入到目标消息队列之后，可能存在需要将目标线程唤醒。这需要分两种情况讨论。

  1. 插入的消息在目标消息队列头部
  2. 插入的消息在目标消息队列中间

对于第一种情况，由于保存的消息队列的头部的消息发生了变化，因此，当前线程就需要将目标线程唤醒，以便它可以对保存在目标消息队列头部的新消息进行处理。但是如果这时候目标线程不是正处于睡眠等待状态，那么当前线程就不需要对它执行唤醒操作。当前正在处理的MessageQueue对象的成员变量mBlocked记录了目标线程是否正处于睡眠等待状态。如果它的值等于true，那么就表示目标线程正处于睡眠等待状态，这时候当前线程就需要将它唤醒。第42行代码将它保存在needWake中，以便当前线程接下来可以决定是否需要将目标线程唤醒。

对于第二种情况，如果情况满足以下三点：1.目标线程正处于睡眠等待状态 2.插入消息是异步消息，3.消息队列头target不存在。那么就需要将目标线程唤醒，这时候将needWake置为true。如果以上三点只要有一点无法满足则不需要执行目标线程唤醒操作,这时候将needWake置为false。具体逻辑看第47行代码。

3：wake

线程消息的的循环过程

消息的循环过程是先于消息的发送过程发生，只不过是启动循环后由于消息队列中没有任何消息而导致了线程休眠。直到等待消息发送，消息队列存在消息之后便会出现之前所说的唤醒操作导致消息循环唤醒并进行执行。 整个的循环调用过程如下图所示：

这个过程分为3个步骤，我们按上图中的调用流程进行分析

1：loop

第9-14行代码获取当前线程的消息队列，第30-31代码会不断循环检查队列中是否存在消息需要进行处理。如果当前队列不存在消息则执行queue.next()后会阻塞。

2: next

我们可以很清楚的看到，next方法里面也是一个循环，而这个循环只有在以下这3种情况才可能退出并返回：

  1. 如果消息循环已经退出并被处理
  2. 存在有新的消息，但消息已经处理并被挂起
  3. 存在有新的消息，消息并未挂起且必须立刻处理

第一种情况执行第9-11行代码逻辑，mPtr值为0，该值指向了底层的消息队列，即表示底层的消息队列已经不存在。这类情况一般发生在重启一个已经退出循环的Looper。

第二种情况执行57-60行代码逻辑，mQuitting为true，表示消息队列正在退出。

第三种情况执行39-49行代码逻辑，也就是说当前消息队列存在信息，并且消息立刻处理，即35行代码执行为false。方法返回之前将当前消息标记为已使用并将当前消息队列的指针往下移。

如果存在有新的消息，消息并未挂起且消息并未到时间处理。这时候则执行第37行代码，计算当前线程处理该消息的时间。由于线程没有新的消息需要处理，并不是立马循环执行nativePollOnce方法进入休眠状态，而是先看看空闲消息处理器是否存在有需要处理的消息（其中pendingIdleHandlerCount用来保存注册到消息队列中的空闲消息处理器（IdleHandler）的个数），如果存在则执行83-99行代码逻辑，先处理完空闲消息后再返回，如果不存在则执行69-71行代码逻辑后返回。

我们可以看出，如果当前消息队列不存在新消息，则nativePollOnce会阻塞，阻塞时间由nextPollTimeoutMillis决定；如果存在新消息则继续往下执行。由此看来nativePollOnce是实现了检测当前是否有新消息的机制。

3: nativePollOnce

nativePollOnce是一个JNI方法,其具体执行逻辑是在Native层，里面的逻辑大体是通过Native层独有的管道通讯方式，阻塞式等待之前消息发送流程中往管道写入消息提醒，一旦收到消息提醒后便会返回。涉及Native层的调用逻辑这里不进行描述，具体在之后的章节再详细进行描述。

线程消息的处理过程

当消息循环获取到新消息后，便会执行消息处理。

这个过程总共分为三个步骤，分别是loop、disptchMessage、handleMessage。

1：loop

第16行代码取到新消息后，通过第17行代码判断消息是否为空，不为空的情况下执行第22行代码，将消息交由target对象的dispatchMessage方法来处理。

2：disptchMessage

该方法按照以下顺序进行逻辑处理：

  1. 如果要处理的消息在发送前指定了一个回调接口，即第8行代码语句为true，第9行代码就会调用handleCallback处理逻辑，再执行第21行代码，最终实际上是执行了callback的run方法。
  2. 如果条件1不满足，并且负责分发消息的Handler对象的成员变量mCallback指向了一个回调接口，即第11行语句为true，那么就会调用这个回调接口的方法 handleMessage来处理消息
  3. 如果条件2不满足，或者负责分发消息的Handler的成员变量mCallback为null,即第行语句为false，则调用目前对象的handleMessage方法处理该消息
